# Chapter 23: Appendix B: File Reference

Appendix

Every file Draft generates, organized by directory. For each file: name, purpose, typical size, audience, and lifecycle.

## draft/ (Root)

Generated by `/draft:init`. These files are the primary context artifacts — the source of truth for architecture, conventions, and project identity. Edit them directly; derived files are regenerated automatically.

| File | Purpose | Typical size | Audience | Lifecycle |
|------|---------|-------------|----------|-----------|
| `index.md` | Plain docs index listing all context files in this directory and their roles. | ~30 lines | Human | Updated on refresh. |
| `architecture.md` | Source of truth. 10-section graph-primary engineering reference with Mermaid diagrams, module boundaries, data flows, API definitions, concurrency model, security architecture, and testing infrastructure. | 800–2,000 lines | Human + AI | Updated on `/draft:init` refresh or `/draft:deep-review` with `--refresh`. Never hand-generate from `.ai-context.md`. |
| `.ai-context.md` | Token-optimized AI context derived from `architecture.md`. 15+ structured sections (META, GRAPH:*, INVARIANTS, INTERFACES, CATALOG:*, THREADS, CONFIG, ERRORS, CONCURRENCY, EXTEND:*, TEST, FILES, VOCAB). Loaded for implementation and quality tasks. | 200–400 lines | AI | Regenerated automatically when `architecture.md` changes. Do not edit directly. |
| `.ai-profile.md` | Ultra-compact always-on profile derived from `.ai-context.md`. Loaded for every interaction regardless of task complexity. Contains: language, framework, database, auth mechanism, API style, critical invariants, safety rules, active tracks, recent changes. | 20–50 lines | AI | Regenerated automatically. |
| `product.md` | Product vision, target users, goals, success criteria, and guidelines. Reviewed by product managers; revisited after OKR cycles or product pivots. | 100–300 lines | Human | Edited by team; persists across refreshes. |
| `tech-stack.md` | Languages, frameworks, libraries, patterns, and accepted design decisions. Auto-detected for brownfield projects. | 100–250 lines | Human + AI | Edited by tech leads; persists across refreshes. |
| `workflow.md` | Team engineering workflow: TDD preference, commit strategy, review checklists, phase verification procedures. Read by every Draft agent. | 100–200 lines | Human + AI | Edited by engineering leads; persists across refreshes. |
| `guardrails.md` | Hard guardrails (human-defined constraints), learned conventions, and learned anti-patterns. `/draft:learn` populates conventions and anti-patterns automatically. | 100–300 lines | Human + AI | Populated by `/draft:learn`; refined manually. |
| `tracks.md` | Index of all tracks in `draft/tracks/`. Lists track name, type, status, and creation date. | ~20 lines | Human | Auto-updated by `/draft:new-track` and `/draft:implement`. |

`architecture.md` is the primary artifact. `.ai-context.md` is derived from it (token-optimized for AI). `.ai-profile.md` is derived from `.ai-context.md` (ultra-compact, always loaded). Edits flow downward: update `architecture.md`, then the derived files are regenerated automatically via the Condensation Subroutine.

## draft/graph/

The knowledge graph directory. Contains only a single committed file — all structural graph data is queried live from the `codebase-memory-mcp` engine (by DeusData, 159 languages, 100% local). No `architecture.json`, `hotspots.jsonl`, or `*.mermaid` files are committed here.

| File | Purpose | Lifecycle |
|------|---------|-----------|
| `schema.yaml` | Gate marker. Contains engine metadata, project identity, and point-of-index counts. The `access: engine-live` field signals that structural data must be queried from the engine, not read from committed files. | Written by `/draft:init`; updated on refresh. |
| `root-link.json` | Present only in sub-module `draft/graph/` directories. Points to the root graph location for cross-module queries. | Written by `/draft:init` when run inside a monorepo sub-module. |

## draft/.state/

Internal state files for incremental refresh, signal classification, and cross-session continuity. Not intended for direct editing.

| File | Purpose |
|------|---------|
| `freshness.json` | SHA-256 hashes of all analyzed source files. Enables incremental refresh — only changed, new, and deleted files are re-analyzed. |
| `signals.json` | Signal classification baseline from Phase 1 discovery. Diffed on refresh to detect structural drift. |
| `run-memory.json` | Run metadata, unresolved questions, and resumable checkpoints. Enables interrupted init/refresh to resume where it left off. |
| `facts.json` | Atomic architectural facts with source file references, confidence scores, and relationship edges. Supports contradiction detection on refresh. |

## draft/tracks/<track-name>/

Each track (feature, fix, or refactor) gets its own directory with these files:

| File | Purpose | Lifecycle |
|------|---------|-----------|
| `spec.md` | Requirements, acceptance criteria, non-goals, and technical approach. Grounded in architecture and tech stack. | Created by `/draft:new-track`; edited by developer before implementation. |
| `plan.md` | Phased task breakdown with dependencies and verification steps. Uses status markers: `[ ]` Pending, `[~]` In Progress, `[x]` Completed, `[!]` Blocked. | Created by `/draft:new-track`; updated by `/draft:implement`. |
| `metadata.json` | Track metadata: name, type, status, creation date, last-modified, and `impact` block (files_touched, modules_touched, downstream_files, by_category). | Created by `/draft:new-track`; updated automatically. |

## Optional Track Files

These files are created by specific commands when run against a track:

| File | Created by | Purpose |
|------|-----------|---------|
| `bughunt-report.md` | `/draft:bughunt` | Severity-ranked bug findings with code evidence. |
| `coverage-report.md` | `/draft:coverage` | Test coverage gap analysis against plan and acceptance criteria. |
| `review-report.md` | `/draft:review` or `/draft:deep-review` | Three-stage review findings. |
| `deploy-checklist.md` | `/draft:deploy-checklist` | Pre-deployment checklist for the track. |

## draft/adrs/

Architecture Decision Records authored by `/draft:adr`. Each ADR is a separate markdown file named `NNN-<slug>.md` following the MADR format. ADRs are referenced from `architecture.md` and `tracks/*/spec.md` as appropriate.
